# 量化格式详解

本文档深入介绍 GGUF 量化格式的工作原理——从位级别的数据布局，到 GPU 着色器如何在矩阵乘法中实时执行反量化。

适合阅读对象：希望理解模型压缩机制的开发者、在选择量化格式时需要做出权衡的用户，以及想要修改 WGSL 着色器的贡献者。

---

## 什么是量化？

大型语言模型的权重通常以 32 位浮点数（F32）存储。对于一个 70 亿参数的模型，这意味着约 **28 GB** 的存储空间——远超大多数消费级 GPU 的显存。

量化通过降低每个权重的位宽来压缩模型。核心思路是：

```
原始 F32 权重（32 位/权重）
         ↓
量化为整数（4 位或 8 位/权重）
         ↓
配以缩放因子（scale）记录原始数值范围
         ↓
在 GPU 着色器中推理时实时还原近似值
```

压缩比达到 **4–8×**，精度损失极小（通常困惑度增加 < 1%）。

## GGUF 量化格式族

### Q4_0 — 简单 4 位量化

最基础的格式，每 **32 个权重**构成一个块，每块一个 F32 缩放因子：

```
块布局（136 字节/块，32 个权重）：
┌─────────────────────────────────────────┐
│  scale (f32, 4 字节)                    │
│  quants[16] (u8 × 16 = 16 字节)         │
│    每个 u8 储存 2 个 4 位整数            │
│    高 4 位 = 偶数权重                    │
│    低 4 位 = 奇数权重                    │
└─────────────────────────────────────────┘
```

反量化公式：
```
value = scale × (quant − 8)
```

其中 `quant` 是 0–15 范围内的 4 位整数，减去 8 使其以零为中心（−8 到 +7）。

**优点**：格式简单，兼容性最好，硬件适应性强  
**缺点**：每块只有一个缩放因子，精度不如 K-quant

---

### Q8_0 — 简单 8 位量化

与 Q4_0 相同的块结构，但每个权重 8 位：

```
块布局（34 字节/块，32 个权重）：
┌─────────────────────────────────────────┐
│  scale (f16, 2 字节)                    │
│  quants[32] (i8 × 32 = 32 字节)         │
└─────────────────────────────────────────┘
```

反量化公式：
```
value = scale × quant
```

`quant` 是有符号 8 位整数（−128 到 +127）。

**优点**：最接近 F32 精度，每块信息损失最少  
**缺点**：文件大小约为 Q4_0 的两倍

---

### K-quant 系列（Q4_K、Q5_K、Q6_K）

K-quant 是更先进的混合精度格式，使用**超级块（superblock）**：每个超级块包含 8 个子块，共 **256 个权重**，并有两级缩放因子：

```
超级块布局（Q4_K_M 为例）：
┌─────────────────────────────────────────────┐
│  超级块缩放因子 d (f16)                      │
│  超级块最小值 dmin (f16)                     │
│  子块缩放因子 scales[12] (6 位压缩存储)       │
│    每个子块有独立的 scale 和 min 值           │
│  量化值 quants[128] (u8, 4 位/权重 × 256)   │
└─────────────────────────────────────────────┘
```

反量化公式（每个子块）：
```
scale_i = d × scales[i]
min_i   = dmin × mins[i]
value   = scale_i × quant − min_i
```

**为什么更好**：每个子块有独立的缩放范围，能更精确地表示权重分布中的局部变化。对于注意力层中权重差异较大的区域，精度提升尤为明显。

**Q4_K_M 中的"M"（Medium）**：混合精度策略——部分层使用 Q6_K（注意力层），其余层使用 Q4_K，在精度和文件大小之间取得平衡。

## GPU 着色器中的实时反量化

Airframe 不采用先反量化再计算的两步流程——权重解码**在矩阵乘法着色器内部实时完成**：

```wgsl
// Q4_0 反量化（WGSL 伪代码）
fn dequant_q4_0(block: ptr<storage, BlockQ4_0>, idx: u32) -> f32 {
    let scale = f32(block.scale);
    let byte_idx = idx / 2u;
    let nibble = block.quants[byte_idx];
    let q = select(
        (nibble >> 4u) & 0xFu,   // 偶数权重：高 4 位
        nibble & 0xFu,             // 奇数权重：低 4 位
        (idx % 2u) == 0u
    );
    return scale * (f32(q) - 8.0);
}
```

每次着色器读取一个权重，就立即将其从量化整数转换为 F32，然后与激活值相乘，结果直接累加到输出。整个过程没有额外的显存分配。

**为什么不预先反量化？**

- 预先反量化的 F32 权重需要的显存是量化版本的 4–8 倍
- 实时反量化将内存带宽压力保持在量化数据的水平
- 现代 GPU 的算术吞吐量远超内存带宽，反量化计算几乎是"免费"的

## 各格式性能对比

以 7B 参数模型为基准：

| 格式 | 文件大小 | 困惑度增量 | 典型用途 |
|------|---------|-----------|--------|
| F32（原始） | 28 GB | 0（基准） | 仅用于对比 |
| Q8_0 | 7.2 GB | +0.1% | 精度优先 |
| Q6_K | 5.5 GB | +0.2% | 高质量 |
| Q5_K_M | 4.8 GB | +0.4% | 质量/大小均衡 |
| **Q4_K_M** | **4.1 GB** | **+0.6%** | **推荐首选** |
| Q4_0 | 3.9 GB | +1.2% | 兼容性优先 |
| Q3_K_M | 3.3 GB | +2.1% | 显存非常紧张时 |

> **注意**：Shimmy 目前不支持 Q2_K、Q3_K 系列及 IQ（imatrix）格式。尝试加载不支持的格式会在启动时报错。BF16 格式也暂不支持。

## 如何选择量化格式

**首选 Q4_K_M**：质量与大小的最佳平衡点，覆盖绝大多数应用场景。

**选 Q8_0 的情形**：
- 在量化敏感任务（代码生成、精确数学）上需要最高精度
- 显存充足（例如 24 GB 显存运行 7B 模型）

**选 Q4_0 的情形**：
- 目标模型只提供 Q4_0 格式
- 需要与 llama.cpp 等其他工具兼容

**避免 Q3 及以下**：
- Airframe 目前不支持
- 即使支持，精度损失也相当明显

## quant_verify 工具

Airframe 内置一个 GPU/CPU 反量化一致性验证工具：

```powershell
$env:LIBSHIMMY_MODEL_PATH = "D:\models\TinyLlama-1.1B-Chat-v1.0.Q4_0.gguf"
cargo run --release --bin quant_verify
```

该工具对模型中的每种量化格式各取样本，分别在 GPU 和 CPU 上执行反量化，然后对比结果。输出示例：

```
Q4_0   [tensor: blk.0.attn_q.weight]   OK  (max_delta=0.000031)
Q4_K   [tensor: blk.0.ffn_gate.weight] OK  (max_delta=0.000015)
```

若出现 `MISMATCH`，说明着色器存在 bug。

> **限制**：显存不足 2 GB 的设备上该工具可能失败（单个绑定缓冲区上限）。
