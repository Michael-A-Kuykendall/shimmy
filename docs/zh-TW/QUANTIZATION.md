# 量化格式詳解

本文件深入介紹 GGUF 量化格式的運作原理——從位元級的資料布局，到 GPU 著色器如何在矩陣乘法中即時執行反量化。

適合閱讀對象：希望理解模型壓縮機制的開發者、在選擇量化格式時需要做出取捨的使用者，以及想要修改 WGSL 著色器的貢獻者。

---

## 什麼是量化？

大型語言模型的權重通常以 32 位元浮點數（F32）儲存。對於一個 70 億參數的模型，這意味著約 **28 GB** 的儲存空間——遠超大多數消費級 GPU 的顯存。

量化透過降低每個權重的位元寬度來壓縮模型。核心思路是：

```
原始 F32 權重（32 位元/權重）
         ↓
量化為整數（4 位元或 8 位元/權重）
         ↓
配以縮放因子（scale）記錄原始數值範圍
         ↓
在 GPU 著色器中推論時即時還原近似值
```

壓縮比達到 **4–8×**，精度損失極小（通常困惑度增加 < 1%）。

## GGUF 量化格式族

### Q4_0 — 簡單 4 位元量化

最基礎的格式，每 **32 個權重**構成一個區塊，每個區塊一個 F32 縮放因子：

```
區塊布局（136 位元組/區塊，32 個權重）：
┌─────────────────────────────────────────┐
│  scale (f32, 4 位元組)                   │
│  quants[16] (u8 × 16 = 16 位元組)        │
│    每個 u8 儲存 2 個 4 位元整數           │
│    高 4 位元 = 偶數權重                   │
│    低 4 位元 = 奇數權重                   │
└─────────────────────────────────────────┘
```

反量化公式：
```
value = scale × (quant − 8)
```

其中 `quant` 是 0–15 範圍內的 4 位元整數，減去 8 使其以零為中心（−8 到 +7）。

**優點**：格式簡單，相容性最佳，硬體適應性強
**缺點**：每個區塊只有一個縮放因子，精度不如 K-quant

---

### Q8_0 — 簡單 8 位元量化

與 Q4_0 相同的區塊結構，但每個權重 8 位元：

```
區塊布局（34 位元組/區塊，32 個權重）：
┌─────────────────────────────────────────┐
│  scale (f16, 2 位元組)                   │
│  quants[32] (i8 × 32 = 32 位元組)        │
└─────────────────────────────────────────┘
```

反量化公式：
```
value = scale × quant
```

`quant` 是有號 8 位元整數（−128 到 +127）。

**優點**：最接近 F32 精度，每個區塊資訊損失最少
**缺點**：檔案大小約為 Q4_0 的兩倍

---

### K-quant 系列（Q4_K、Q5_K、Q6_K）

K-quant 是更先進的混合精度格式，使用**超級區塊（superblock）**：每個超級區塊包含 8 個子區塊，共 **256 個權重**，並有兩級縮放因子：

```
超級區塊布局（Q4_K_M 為例）：
┌─────────────────────────────────────────────┐
│  超級區塊縮放因子 d (f16)                    │
│  超級區塊最小值 dmin (f16)                   │
│  子區塊縮放因子 scales[12]（6 位元壓縮儲存） │
│    每個子區塊有獨立的 scale 和 min 值         │
│  量化值 quants[128]（u8，4 位元/權重 × 256） │
└─────────────────────────────────────────────┘
```

反量化公式（每個子區塊）：
```
scale_i = d × scales[i]
min_i   = dmin × mins[i]
value   = scale_i × quant − min_i
```

**為什麼更好**：每個子區塊有獨立的縮放範圍，能更精確地表示權重分布中的局部變化。對於注意力層中權重差異較大的區域，精度提升尤為明顯。

**Q4_K_M 中的「M」（Medium）**：混合精度策略——部分層使用 Q6_K（注意力層），其餘層使用 Q4_K，在精度和檔案大小之間取得平衡。

## GPU 著色器中的即時反量化

Airframe 不採用先反量化再計算的兩步流程——權重解碼**在矩陣乘法著色器內部即時完成**：

```wgsl
// Q4_0 反量化（WGSL 虛擬碼）
fn dequant_q4_0(block: ptr<storage, BlockQ4_0>, idx: u32) -> f32 {
    let scale = f32(block.scale);
    let byte_idx = idx / 2u;
    let nibble = block.quants[byte_idx];
    let q = select(
        (nibble >> 4u) & 0xFu,   // 偶數權重：高 4 位元
        nibble & 0xFu,             // 奇數權重：低 4 位元
        (idx % 2u) == 0u
    );
    return scale * (f32(q) - 8.0);
}
```

每次著色器讀取一個權重，就立即將其從量化整數轉換為 F32，然後與激活值相乘，結果直接累加到輸出。整個過程沒有額外的顯存分配。

## 各格式效能對比

以 7B 參數模型為基準：

| 格式 | 檔案大小 | 困惑度增量 | 典型用途 |
|------|---------|-----------|--------|
| F32（原始） | 28 GB | 0（基準） | 僅用於對比 |
| Q8_0 | 7.2 GB | +0.1% | 精度優先 |
| Q6_K | 5.5 GB | +0.2% | 高品質 |
| Q5_K_M | 4.8 GB | +0.4% | 品質/大小均衡 |
| **Q4_K_M** | **4.1 GB** | **+0.6%** | **推薦首選** |
| Q4_0 | 3.9 GB | +1.2% | 相容性優先 |
| Q3_K_M | 3.3 GB | +2.1% | 顯存非常緊張時 |

> **注意**：Shimmy 目前不支援 Q2_K、Q3_K 系列及 IQ（imatrix）格式。嘗試載入不支援的格式會在啟動時報錯。BF16 格式也暫不支援。

## 如何選擇量化格式

**首選 Q4_K_M**：品質與大小的最佳平衡點，涵蓋絕大多數應用場景。

**選 Q8_0 的情形**：
- 在量化敏感任務（程式碼生成、精確數學）上需要最高精度
- 顯存充足（例如 24 GB 顯存執行 7B 模型）

**選 Q4_0 的情形**：
- 目標模型只提供 Q4_0 格式
- 需要與 llama.cpp 等其他工具相容

## quant_verify 工具

Airframe 內建一個 GPU/CPU 反量化一致性驗證工具：

```powershell
$env:LIBSHIMMY_MODEL_PATH = "D:\models\TinyLlama-1.1B-Chat-v1.0.Q4_0.gguf"
cargo run --release --bin quant_verify
```

該工具對模型中的每種量化格式各取樣本，分別在 GPU 和 CPU 上執行反量化，然後對比結果。若出現 `MISMATCH`，代表著色器存在問題。
